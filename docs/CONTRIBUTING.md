## Процесс работы

Основная ветка разработки: `main`.

### 1. Подготовьте репозиторий

```bash
# Клонировать репозиторий
git clone https://github.com/GrimStack-Education/spotWave.git

# Перейти в директорию проекта
cd spotWave

# Установить зависимости
pnpm install
```

### 2. Создайте отдельную ветку

```bash
# Создать ветку от main
git checkout -b feature/short-description
```

Допустимые префиксы веток:

<div align="center">

| **Префикс**  | **Назначение**                      |
| :----------- | :---------------------------------- |
| `feature/*`  | Новая функциональность              |
| `fix/*`      | Исправление ошибок                  |
| `docs/*`     | Изменения в документации            |
| `refactor/*` | Рефакторинг без изменения поведения |
| `test/*`     | Тесты и тестовая инфраструктура     |
| `chore/*`    | Служебные задачи                    |

</div>

### 3. Держите ветку актуальной

```bash
# Получить актуальные изменения из origin
git fetch origin

# Перебазировать текущую ветку на main
git rebase origin/main
```

## Требования к изменениям

- Один Pull Request должен решать одну задачу.
- Изменения должны быть локальными и понятными по объему.
- Новая функциональность должна сопровождаться тестами.
- Если поведение изменено, документация должна быть обновлена.
- Нельзя добавлять секреты, токены, пароли и приватные ключи.

### Краткие ожидания по коду

<div align="center">

| **Стек**    | **Ожидание**                                                              |
| :---------- | :------------------------------------------------------------------------ |
| TypeScript  | Строгая типизация и предсказуемые контракты                                |
| React       | Компоненты с понятными пропсами и без лишних side effects                  |
| Backend     | Предсказуемые API-контракты и аккуратные изменения логики                 |
| Prisma      | Миграции и изменения схемы должны быть понятными и проверяемыми           |
| Docker      | Локальные compose-конфигурации должны быть воспроизводимыми               |

</div>

## Тестирование изменений

Перед PR убедитесь, что:

- тесты проходят локально;
- линтеры и проверки стиля проходят;
- сборка проекта успешна;
- изменение покрыто тестами на уровне, достаточном для регрессий.

### Минимальный набор команд перед PR

```bash
# Из корня репозитория
pnpm format:check
pnpm lint:fix
pnpm build
```

### Точечные проверки

```bash
# Frontend
pnpm dev:frontend
pnpm build:frontend

# Backend
pnpm dev:backend
pnpm build:backend
```

### Docker-проверка

```bash
# Из корня репозитория
docker compose -f infrastructure/docker/docker-compose.yml up --build
```

Если есть ошибки форматирования или сборки, исправьте их до создания PR.

## Pull Requests

### Чек-лист перед открытием PR

- [ ] Ветка создана от актуального `main`
- [ ] Лишние/временные файлы не добавлены
- [ ] Добавлены или обновлены тесты
- [ ] Обновлена документация при изменении поведения
- [ ] Изменения проходят локальные проверки
- [ ] `pnpm format:check` и `pnpm build` проходят без ошибок

### Шаблон описания PR

```markdown
## Описание

Что было сделано и почему

## Type of change

- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Как проверялось

Кратко опишите сценарии проверки

## Чек-лист

- [ ] Код соответствует стилю проекта
- [ ] Добавлены нужные тесты
- [ ] Обновлена документация
- [ ] Все проверки проходят

## Related issue

Closes #123
```

### После открытия PR

1. Автопроверки CI должны быть зелёными.
2. Требуется минимум одно ревью и одобрение.
3. Замечания ревью должны быть закрыты до merge.

### Что проверяет CI

- Frontend: `pnpm build`
- Backend: сборка и базовые проверки запуска
- Docker: воспроизводимость контейнерной сборки

## Commit сообщения

Используйте формат Conventional Commits:

```text
<type>(<scope>): <subject>
```

Рекомендуемые type:

<div align="center">

| **Type**   | **Назначение**                      |
| :--------- | :---------------------------------- |
| `feat`     | Новая функциональность              |
| `fix`      | Исправление ошибки                  |
| `docs`     | Изменения в документации            |
| `refactor` | Рефакторинг без изменения поведения |
| `test`     | Добавление или обновление тестов    |
| `chore`    | Служебные изменения и обслуживание  |
| `perf`     | Оптимизация производительности      |
| `style`    | Форматирование и stylistic-правки   |

</div>

Рекомендуемые scope:

<div align="center">

| **Scope**    | **Примеры**                              |
| :----------- | :--------------------------------------- |
| `app`        | Роуты, инициализация, провайдеры         |
| `pages`      | Страницы (Home и другие экраны)          |
| `components` | UI-компоненты                            |
| `features`   | Сценарии и пользовательские флоу        |
| `entities`   | Сущности домена                         |
| `shared`     | Общие модули (api, ui, lib)             |
| `db`         | Prisma schema и миграции                |
| `docker`     | Compose, Dockerfile и связанные файлы   |

</div>

## Коммуникация

- Если задача требует изменения поведения, коротко опишите, что меняется и почему.
- Если что-то затрагивает инфраструктуру, укажите, как это проверить локально.
- Если есть спорный trade-off, зафиксируйте его в PR description.
